# N-gram Language Model

A dependency-free (pure Python standard library) statistical language model:
unigram through 4-gram models with **add-k smoothing** and **recursive linear
interpolation**, trained on *Alice's Adventures in Wonderland*, with the
interpolation weight tuned by grid search against a **held-out validation
set** — not hand-picked. Evaluated with perplexity on train/val/test splits
plus an out-of-domain corpus (*The Adventures of Sherlock Holmes*), and used
to generate sample text.

No numpy, no scikit-learn, no PyTorch — just counting, probability, and
Python's `random` module. The point of this project is the language-modeling
math and evaluation methodology, done rigorously with the simplest possible
model class.

## Why a from-scratch n-gram model in 2026?

Transformer-based LLMs are the state of the art, but the n-gram model is
where the *core ideas* of statistical language modeling are visible without a
GPU or a framework: what a probability distribution over the next word
actually looks like, why smoothing exists, why perplexity is the standard
metric, and how easy it is to accidentally fool yourself with an evaluation
setup that "generalizes" for the wrong reason (see [Findings](#findings)
below — this project found exactly that kind of gotcha).

## Method

**Tokenization** (`ngram/tokenizer.py`): regex sentence-splitting, lowercase
word tokens (letters plus internal apostrophes/hyphens: `don't`, `well-known`).

**Vocabulary** (`ngram/corpus.py`): built from the *training split only*.
Words appearing fewer than `min_count` times in training are folded into a
single `<unk>` token — including in training itself, not just at test time.
This matters: if every training word were kept in-vocabulary, the model would
never actually experience an unknown word during training, and its `<unk>`
probability estimate would be meaningless at test time. `</s>` (end of
sentence) is a real vocabulary member the model can predict; `<s>`
(start-of-sentence padding) is context-only and is never itself a prediction
target.

**Model** (`ngram/model.py`): for order *n*, probability is computed by
recursively interpolating down to the unigram base case:

```
P_1(w)          = (count(w) + k) / (N + k·|V|)                      [add-k unigram]
P_n(w | ctx)    = λ · P̂_n(w | ctx) + (1 − λ) · P_{n−1}(w | ctx[1:])
P̂_n(w | ctx)    = (count(ctx, w) + k) / (count(ctx) + k·|V|)         [add-k n-gram MLE]
```

`k=1.0` (Laplace/add-one) throughout. `λ` is a single scalar re-used at every
recursion level for a given top order — a simplification of full
Jelinek-Mercer interpolation (which fits a separate λ per level); documented
here as a limitation, not hidden. `λ` for n=2, 3, 4 is chosen by **grid search
over {0.1, ..., 0.9} to minimize validation-set perplexity** (`train.py`).

**Evaluation:** corpus-level perplexity, `2^(−1/N · Σ log2 P(w_i | context))`,
computed over a proper 80/10/10 train/validation/test split (774/96/98
sentences, seed 42) plus a fully separate out-of-domain corpus.

## Results

Trained on 968 sentences / 26,500 tokens of *Alice's Adventures in
Wonderland*; vocabulary 1,303 tokens (`min_count=2`).

| n | tuned λ | train PPL | val PPL | test PPL | out-of-domain PPL (Sherlock Holmes) |
|---|---|---|---|---|---|
| 1 (unigram) | n/a | 265.49 | 228.43 | 217.54 | 139.70 |
| **2 (bigram)** | **0.3** | **213.81** | **211.94** | **198.03** | 144.30 |
| 3 (trigram) | 0.1 | 232.42 | 222.99 | 210.69 | 146.35 |
| 4 (4-gram) | 0.1 | 234.35 | 233.09 | 220.05 | 154.91 |

## Findings

**1. Bigram wins — data sparsity, not model capacity, is the bottleneck.**
Trigram and 4-gram both get *tuned toward the bigram* (λ = 0.1, meaning only
10% weight on the higher-order term) because the grid search is directly
optimizing validation perplexity, and with only ~21K training tokens there
isn't enough data to trust 3- and 4-word context counts — most contexts of
length 3+ are seen once or never. The higher-order models end up performing
*worse* than the plain bigram on held-out test data (210.7 and 220.1 vs.
198.0 PPL) despite having strictly more information available, a textbook
illustration of why "more context" doesn't automatically mean "better model"
without enough data to estimate it. Deep learning solves this with continuous
embeddings that let similar contexts share statistical strength — this
project's n-gram approach makes it obvious *why* that matters.

**2. Out-of-domain perplexity looked deceptively good — until we checked
why.** The initial expectation was that Sherlock Holmes (crime fiction) would
look much harder to predict than held-out Alice text (fantasy) — different
vocabulary, different subject matter. The raw numbers don't show that: the
bigram model's out-of-domain perplexity (144.3) isn't far from its in-domain
test perplexity (198.0), and for the unigram model out-of-domain PPL is
*lower* (139.7 vs. 217.5). Splitting perplexity by whether the target token
was in-vocabulary or `<unk>` explains it
([diagnostic table](reports/RESULTS.md#oov-diagnostic-does-out-of-domain-perplexity-reflect-real-domain-shift-or-is-it-masked-by-unk-bucketing)):
Sherlock Holmes has more than **twice** the out-of-vocabulary rate of the
Alice test set (21.2% vs. 9.7% of tokens), but every one of those unknown
words gets bucketed into the same `<unk>` token — and `<unk>` itself is a
comparatively *easy*, well-estimated prediction (perplexity ~25 at `<unk>`
positions in both sets, vs. ~240+ for known words in both sets). **The
model isn't generalizing better to Sherlock Holmes; the `<unk>`-bucketing
scheme is hiding a larger domain gap by converting "I have no idea what word
this is" into "I confidently predict `<unk>`."** This is a known, real
failure mode of naive OOV handling in n-gram models, and a big part of why
modern language models use subword/BPE tokenization instead of a single
catch-all unknown-word token: it removes exactly this kind of blind spot.

**3. Generated text is locally plausible, globally incoherent** — exactly
what an n-gram model should produce. Phrases like *"who what \<unk> \<unk>
there stick said falling out out of and in her size you when never is
directions"* (bigram, see [full samples](reports/RESULTS.md#sample-generated-text-temperature09))
show short local patterns the model actually learned (e.g. "said the",
repeated character names) glued together with no long-range structure,
because the model has no way to represent anything beyond a 1-3 word window.

## Usage

```bash
python3 train.py                       # full pipeline: train, tune, evaluate, report
python3 -m pytest test_ngram.py -v     # or: python3 test_ngram.py
python3 generate.py --order 2 --num 5 --temperature 0.9
python3 generate.py --corpus data/sherlock_holmes.txt --order 3 --seed 7
```

## Project structure

```
ngram/
  tokenizer.py    sentence splitting + word tokenization
  corpus.py       vocabulary building, OOV replacement, boundary padding
  model.py        NGramModel: add-k smoothing, recursive interpolation,
                  perplexity, sampling/generation
train.py          full pipeline: load -> split -> tune lambda -> evaluate -> report
generate.py       standalone text-generation CLI
test_ngram.py     19 unit + integration tests
data/             alice_wonderland.txt, sherlock_holmes.txt (public domain,
                  see CITATION.txt), cleaned of Project Gutenberg boilerplate
reports/          RESULTS.md (regenerated each run of train.py)
```

## Testing

`python3 test_ngram.py` — 19 tests, all passing: tokenizer edge cases
(apostrophes, hyphens, empty sentences), vocabulary/OOV/boundary-padding
correctness, probability distributions summing to 1 over the vocabulary at
every model order, add-k smoothing assigning nonzero probability to unseen
n-grams, a model assigning much lower perplexity to its own training
distribution than to random noise tokens, deterministic generation for a
fixed seed, and correct fallback of never-seen query-time words to `<unk>`.

## Limitations

- Single λ per top-order model (not per-level Jelinek-Mercer weights).
- Regex sentence splitter mis-handles abbreviations (e.g. "Mr.") — acceptable
  for this corpus's prose style, not production-grade.
- `<unk>`-bucket OOV handling, with the masking effect documented above in
  Finding 2 — a subword tokenizer would not have this blind spot.
- No smoothing method comparison (e.g. Kneser-Ney, which is known to
  outperform add-k/Laplace on real corpora) — a natural next step.
